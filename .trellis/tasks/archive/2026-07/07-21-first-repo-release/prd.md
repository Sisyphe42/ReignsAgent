# Prepare first repository release

## Goal

Make the first public ReignsAgent repository release reproducible, legally complete, integrity-verifiable, and publishable from a `v0.1.0` tag without changing the delivered Creator or player product scope.

## Background

- PR #25 merged into `master` as `faa09af`, and its Linux CI, Hosted PWA, deployable-player, and Electron source smoke checks were green.
- The post-merge `master` run `29801670644` exposed a remaining Hosted rail-geometry flake: the floating rail icon moved vertically by 5 px even after the test observed `scrollTop === 0`. `master` must be made reliably green before release preparation proceeds.
- The repository has no tags, GitHub Releases, or completed Desktop Artifacts workflow runs.
- Root and most workspace manifests report `0.1.0`, while `apps/desktop-electron/package.json` reports `0.1.2`.
- The Node Creator ZIP omits the ReignsAgent MIT license. Electron's generated `LICENSE` covers Electron, not this project.
- `.github/workflows/desktop.yml` builds four portable Electron ZIPs for manual runs and `v*` tags but only stores temporary workflow artifacts; it does not create a GitHub Release or build the cross-platform Node Creator ZIP.
- `npm audit --omit=dev` reports no runtime vulnerabilities. The full audit reports build-tool findings through Electron Forge's `@electron/rebuild` / `tar` chain and requires explicit remediation or documented acceptance before producing public binaries.
- Signing, notarization, installers, automatic updates, and a bundled WebView2 Runtime are already documented non-goals for the first release.

## Requirements

### R0 — Restore a reliable default-branch gate

- Fix the remaining Hosted floating-rail geometry race without weakening the strict sub-pixel icon-stability requirement.
- Wait for the actual layout/transition/scroll-anchoring state that makes the geometry stable; a momentary `scrollTop === 0` observation is insufficient.
- Reproduce the failure under repeated single-worker Hosted runs, verify the fix repeatedly, and require a green `master` CI run.
- Keep this stabilization in the release-preparation PR so no release commit can bypass the repaired gate.

### R1 — Canonical release identity

- Use `0.1.0` as the canonical product version and `v0.1.0` as the first public tag.
- Keep root, Electron, generated distribution metadata, About/version presentation, archive names, and tag validation consistent.
- Add an automated release-metadata check that fails on version drift or a tag/version mismatch.

### R2 — Legal and attribution completeness

- Include the ReignsAgent MIT license in the Node Creator ZIP and every Electron Creator ZIP under an unambiguous project-owned filename.
- Preserve Electron/Chromium notices and existing sample asset attribution.
- Add a third-party notices artifact or generated notice inventory for bundled frontend/runtime dependencies.
- Extend artifact verification so missing project license or attribution files fail the build.

### R3 — Safe release artifacts

- Produce exactly these primary assets: cross-platform Node Creator ZIP, Windows x64 Electron ZIP, macOS x64 Electron ZIP, macOS arm64 Electron ZIP, Linux x64 Electron ZIP, and `SHA256SUMS.txt`.
- Do not publish a generic Project player EXE: Project EXEs are authored outputs generated by supported local Creator distributions.
- Verify archive contents, filenames, version, executable/runtime presence, absence of `ReignsAgentData`, absence of development/secret files, and SHA-256 integrity.
- Smoke-test packaged artifacts before upload, including the real Windows WebView2 player host path.

### R4 — Release automation

- Preserve `workflow_dispatch` as a non-publishing cross-platform validation path.
- On a matching `v*` tag, build the Node ZIP and all four native Electron ZIPs, download them into one release job, verify the complete asset set, generate checksums, and create a GitHub Release.
- Grant `contents: write` only to the tag-gated publication job; build and validation jobs remain read-only.
- A failed platform build, smoke test, metadata check, or asset verification must prevent publication.
- Re-running a tag workflow must not silently replace mismatched assets.

### R5 — Dependency and repository hygiene

- Triage the full npm audit. Remediate critical/high findings in the executed release toolchain where a compatible fix exists; otherwise record the exact dev-only exposure and upstream constraint in the release review.
- Keep the shipped/runtime dependency audit at zero known vulnerabilities.
- Add `SECURITY.md` with a private reporting route and supported-version statement.
- Add a concise `CHANGELOG.md` / `v0.1.0` release-notes source covering capabilities, platform requirements, and known unsigned/WebView2 limitations.
- Populate the GitHub repository description, homepage, and topics, and enable available secret scanning/push protection and dependency security updates before announcing the release.

### R6 — Final release gate

- Merge this release-preparation work through a dedicated review PR.
- Require a green `master` CI run after merge.
- Run Desktop Artifacts successfully on the final release commit before tagging.
- Tag only the verified `master` commit.
- Inspect the resulting GitHub Release and download/smoke-test representative published assets before public announcement.
- Creating the tag or publishing the GitHub Release remains a separately authorized external action; implementation and dry-run validation must not publish automatically.

## Acceptance Criteria

- [ ] The Hosted navigation-density test passes repeated stress runs and the post-merge `master` CI is green.
- [ ] All product and archive versions resolve to `0.1.0`, and mismatch tests fail deterministically.
- [ ] Node and Electron ZIP verifiers prove that the project license, required attribution, expected runtime files, and no portable user-data directory are present.
- [ ] `npm run verify`, Hosted tests, deployable-player smoke, Windows player smoke, and packaged desktop smoke pass for their supported environments.
- [ ] A manual Desktop Artifacts run succeeds for Windows x64, macOS x64, macOS arm64, and Linux x64 without publishing a release.
- [ ] A dry-run release assembly produces the five expected ZIPs plus `SHA256SUMS.txt`, with every checksum verified.
- [ ] Runtime audit is clean and release-tool audit findings are either fixed or explicitly reviewed as dev-only upstream constraints.
- [ ] Release notes, security policy, repository metadata, and documented first-release limitations are complete.
- [ ] The release workflow publishes only for a version-matching `v*` tag and cannot publish partial or unverified assets.
- [ ] Final tagging/publication is performed only after explicit user approval.

## Out of Scope

- Code signing, Apple notarization, installers, stores, package registries, automatic updates, or bundled WebView2.
- Publishing individual npm workspace packages.
- Publishing Hosted Creator infrastructure.
- New Creator/player features or content-schema changes.
